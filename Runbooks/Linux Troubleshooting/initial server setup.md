### Connection

Изначально, когда голая виртуалка поднята, на неё происходит заход под root, через ssh по паролю.

Подключаемся по ssh с паролем.

На своём ПК генерируем ssh-ключ `ssh-keygen -t ed25519 -c "your@mail"`. Это понадобится дальше, для повышенной безопасности сервера.

### Users initialization
Далее, первый шаг в поднятии сервера - необходимо создать роли на нём. Вспоминаем принцип [Least Privilege](Least%20Privilege).

Создание [разных](Пользователи) ролей, с разными павами - необходимо для разграничения работы, а так же, чтобы можно было отследить, что и где сделал. 

В целях безопасности, после создания ролей, необходимо передать свой .pub ssh ключ на сервак - это альтернативное подключение, чем подключение по паролю.
*Главная проблема в подключении по ssh через пароль - в том, что мы оставляем целый вектор атаки на сервак. Пароль подобрать часто бывает несложно. Подобрать публичный ключ - невозможно.* Поэтому, подключение по паролю через ssh мы [исключим](SSH%20Hardening).

[Передаём ключ ssh на сервер.](Команды%20SSH)

После создания роли для своей работы на сервере, необходимо выдать себе на неё [sudo](sudo) доступ. Далее, мы закроем авторизацию под root.

После того, как мы:
- создали всех необходимы пользователей (хотя, стоит лучше полностью инициализировать свою роль в системе, а дальше уже под ней всё делать).
- передали ключи ssh на юзера для подключения по ключу.
- дали sudo доступ.
- проверили, что всё действительно работает

*Простой способ проверить, работает ли суда под юзером: sudo whoami >> root.*
